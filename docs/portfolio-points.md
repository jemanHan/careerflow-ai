# Portfolio Points

## 제품 관점
- 채용 지원 워크플로우를 실제 사용자 문제(문서 품질/정합성) 중심으로 구조화
- **지원 적합도 분석 → 대화형 보완 → 문서 초안 → 면접 대비 리포트** 순으로 가이드형 흐름을 노출(챗봇이 아닌 단계형)
- 갭 분석 기반 **휴리스틱 적합도 점수(참고 지표)**와 보완 후 재계산으로 개선 루프를 제품 언어로 표현

## 기술 관점
- 단일 프롬프트가 아닌 LangChain 모듈형 체인 설계
- NestJS + Prisma + PostgreSQL로 E2E 저장/조회 흐름 구현
- `WorkflowRun`으로 단계별 AI 실행 증거 로그 저장
- Next.js 최소 UI에서 입력/분석/후속질문/문서/면접질문 플로우 직접 검증 가능
- 실행 중 장애(`npm` 미인식) 원인/대응 문서화
- 단계별 모델 라우팅(기본/고품질)으로 비용-품질 균형 최적화
- 레이트 리미팅/동시실행 락/중복 재실행 스킵으로 과도한 API 사용 방지
- 런타임 안전 운영 규칙 문서화(`docs/cursor-runtime-rules.md`): 포트 선점 확인, 중복 서버 방지, 원인 진단 후 재시도
- 외부 LLM quota 실패(429) 시 fallback으로 워크플로우 연속성을 유지하는 복원력 설계
- 선택 입력(강조 프로젝트)을 실제 우선 컨텍스트로 연결해 분석/생성 품질에 반영
- Gemini free-tier 친화 **역할 기반** 모델 라우팅: 경량(`gemini-3.1-flash-lite` 등) vs 고품질(`gemini-2.5-flash`) — 추출·갭·면접은 light, 문서·리라이트는 quality로 분리(한 모델로 합치지 않음)
- 단계별 워크플로우 로그에 provider/model 메타데이터를 남겨 추적 가능성 확보
- `llmExecution` 메타데이터(`fallbackUsed`, `fallbackReason`, `hasProviderApiKey`)로 장애 원인을 정량 추적
- 레이트리밋/락/재사용 스킵 조합으로 API 과호출과 중복 실행을 운영 레벨에서 제어
- 면접 질문을 단순 리스트가 아닌 **면접 준비 카드**(질문·생성 이유·답변 포인트·과장 주의)로 구조화해 실사용성을 강화
- 문서 생성과 면접 리포트 저장을 분리 병합해, 한 단계 재실행이 다른 단계 결과를 덮어쓰지 않도록 안정화

## 현재 구현됨 (증거 기반)
- 워크플로우 입력/출력/중간 분석 결과가 PostgreSQL(`Application`)에 저장됨
- `fitAnalysisJson`에 AI 추정 적합도(비보장)·강점/약점/추천 보완·보완 전후 점수 델타를 저장
- 단계별 실행 로그가 PostgreSQL(`WorkflowRun`)에 저장됨
- 재실행 제어(`force`/재사용 스킵/락/레이트리밋)로 불필요한 호출을 줄임
- 후속답변 제출 후 후보자 프로필 재분석 흐름이 동작함
- 면접 리포트 `whyAsked`가 JD/이력서/강조 프로젝트/갭 근거를 연결해 질문 의도를 설명함

## 향후 확장(계획)
- 로그인/개인화(`userId` 기반 소유권) 추가
- 저장 우선(save-first) + 버전 기반 수정/부분 재생성
- 반복 개선 UX를 위한 draft history / diff 뷰 추가

## 성능/운영 지표
- 최근 핵심 7단계(분석~생성) 실행 기준 fallback **0회**, 성공 **7/7**
- 샘플 최근 30개 LLM 단계 실행에서 fallback 원인 분류 데이터 확보(키/쿼터/파싱)
- 상세 수치와 해석은 `docs/performance-metrics.md`에 기록

## 인터뷰 어필 포인트
- “왜 이 구조가 실무에서 운영 가능한가”를 설명 가능한 코드/문서로 제시
- 중복 아키텍처(`llm` vs `langchain`)를 제거하고 단일 오케스트레이션으로 정리한 리팩터링 근거
- "모델 라우팅 + 과호출 방지"를 통해 AI 제품의 비용/안정성 문제를 실무적으로 다룬 경험
- "선택 입력도 실질 가치로 연결"되는 제품 설계(라벨 변경 + 우선 컨텍스트 반영 + 결과 UI 증거)
- "모델 라우팅 + 실행 로그 추적"으로 AI 기능의 비용/품질/운영 가시성을 함께 설계한 경험

## 향후 확장 설명 (포트폴리오용)
- 로그인/인증은 현재 MVP 범위에서 의도적으로 제외(개인 개발·단기 검증 우선)했지만, 이는 미구현이 아니라 우선순위 결정의 결과임.
- 이미 PostgreSQL의 `Application`/`WorkflowRun`으로 입력·보완·생성·실행 로그가 저장되므로, `userId` 연결만으로 사용자별 개인화 저장형 제품으로 확장 가능.
- save-first/부분 재생성 전략은 반복 취업 준비 UX에 적합하고, 매번 전체 LLM 호출을 줄여 장기적으로 비용 통제와 운영 수익성 설계에 유리.
