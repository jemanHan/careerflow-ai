# 이 프로젝트에서의 LangChain

## 1. 왜 LangChain을 썼는가

CareerFlow AI의 핵심은 결과를 한 번에 생성하는 것이 아니라,  
이력서·포트폴리오·채용공고를 비교하고 보완한 뒤 문서와 면접 준비까지 이어지는 단계형 워크플로우를 안정적으로 실행하는 데 있습니다.

이 흐름을 단일 호출로 처리하면 아래 문제가 생기기 쉽습니다.

- 단계별 책임이 섞여 수정 범위가 커짐
- 중간 결과를 재사용하거나 재검증하기 어려움
- 실패 지점과 fallback 원인을 분리해 추적하기 어려움
- 작업 성격에 따라 모델을 나누어 쓰기 어려움

LangChain은 프롬프트, 파서, 실행 단위를 단계별로 분리해  
이 워크플로우를 서비스 구조에 맞게 운영할 수 있도록 도와주는 역할을 합니다.

---

## 2. 실제 적용 방식

- `PromptTemplate`
- `RunnableSequence`
- `JsonOutputParser`
- Zod 기반 구조 검증

적용 단계:

- 후보 정보 추출
- 채용공고 정보 추출
- 갭 분석
- 보완 질문 생성
- 문서 초안 2종 생성
- 공고 맞춤 리라이트
- 면접 리포트 생성

실행 진단 정보는 `WorkflowRun`, `llmExecution`에 남기고,  
`LangchainWorkflowService`에서 단계별 라우팅과 fallback 제어를 담당합니다.

---

## 3. 사용자 기준 흐름

1. 입력 저장  
2. 후보·채용공고 구조화 추출  
3. 갭 분석과 `fitAnalysisJson` 생성  
4. 보완 질문 생성 → 답변 반영 → 재분석  
5. 문서 초안 생성 (`coverLetter`, `careerDescription`)  
6. 면접 리포트 생성 (`whyAsked`, `answerPoints` 포함)  

문서 생성이 실패한 경우에도 내부 디버그용 문자열을 그대로 사용자에게 보여주지 않고,  
API 오류와 실행 로그를 기준으로 처리합니다.

---

## 4. 이 프로젝트에서 LangChain이 담당하는 역할

### 단계 분리
분석, 보완, 문서 생성, 면접 준비를 각각 분리해  
어떤 단계가 느린지, 어떤 단계가 실패했는지, 어느 부분을 다시 실행해야 하는지 명확하게 볼 수 있습니다.

### 구조화 출력
`JsonOutputParser`와 Zod 검증을 조합해  
LLM 응답을 그대로 쓰지 않고 프론트와 백엔드가 합의한 구조 안으로 정리합니다.

### 모델 라우팅
작업 성격에 따라 light / quality / premium 슬롯을 나눠  
비용과 품질을 동시에 관리합니다.

### 운영 추적
라우트, 모델명, fallback 여부, 실행 결과를 기록해  
운영 중 발생하는 문제를 실행 단위에서 추적할 수 있습니다.

---

## 5. 적용 범위

현재 LangChain은 문서 입력 비교와 단계형 생성 흐름에 집중해 적용되어 있습니다.  
핵심은 검색 기능 자체가 아니라, 이미 입력된 자료를 기준으로 공고 적합도와 문서 보완 흐름을 안정적으로 연결하는 데 있습니다.

추가 검색 계층이나 더 복잡한 에이전트형 흐름이 필요해지면,  
지금의 단계 구조와 실행 로그 체계를 바탕으로 확장할 수 있습니다.
