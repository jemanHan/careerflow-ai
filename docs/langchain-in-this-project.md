# 이 프로젝트에서의 LangChain

## 1) 왜 LangChain인가

목표는 one-shot 생성이 아니라 **이력서·포트폴리오·채용공고를 비교하고 보완한 뒤 문서와 면접 준비까지 이어지는 단계형 워크플로**입니다.

단일 호출만 쓰면 단계 추적·중간 개선·실패 지점 분리·단계별 모델 라우팅을 일관되게 유지하기 어렵습니다. LangChain으로 **프롬프트·파서·체인 실행**을 단위별로 나눕니다.

---

## 2) 실제로 하는 일

- `PromptTemplate` + `RunnableSequence` + `JsonOutputParser` (+ Zod 검증)  
- 단계별 체인: 후보 추출, JD 추출, 갭 탐지, 보완 질문, **문서 2종 생성**, 리라이트, 면접 리포트  
- `LangchainWorkflowService`에서 **light / quality / (선택) premium** 라우팅  
- 실행 진단은 `WorkflowRun`과 `llmExecution`에 남김  

---

## 3) 사용자 관점 흐름

1. 입력 저장  
2. 구조화 추출(후보·JD)  
3. 갭 분석 + `fitAnalysisJson`  
4. 보완 질문 → 답변 반영 → 재분석  
5. 문서 초안(지원동기/자기소개, 경력기술서)  
6. 면접 리포트(핵심/심화, `whyAsked`·`answerPoints` 등)  

**문서 생성 실패 시**: 내부 디버그용 텍스트를 사용자에게 넣지 않고, API는 오류 응답·로그로 처리합니다.

---

## 4) 이 프로젝트가 “아닌 것” (과장 방지)

| 구분 | 상태 |
|------|------|
| 완전한 RAG(벡터 검색·주입 파이프라인) | **미구현** |
| 자율 멀티툴 에이전트 런타임 | **미구현** |

현재는 **입력 문서 비교 중심의 구조화 파이프라인**에 가깝습니다. 검색·에이전트는 향후 확장으로 문서에서 별도 표기합니다.

---

## 5) LangChain을 쓰는 이유 (요약)

- 단계 책임 분리 → 유지보수·프롬프트 실험 용이  
- 실패 지점 특정 → 운영 디버깅  
- 포트폴리오·면접에서 설계 의사결정 설명이 쉬움  

---

## 6) 포트폴리오용 한 줄 설명 예시

- **짧게**: “LangChain으로 추출·갭분석·보완·문서/면접까지 단계형 체인으로 나누고, 단계별 로그와 모델 라우팅으로 운영 가능하게 만들었습니다.”  
- **조금 더**: “`PromptTemplate + RunnableSequence + JsonOutputParser + Zod`로 체인을 모듈화하고, `WorkflowRun`에 라우트·실행 메타데이터를 남겼습니다. 문서는 2종 필드로 구조화해 프론트와 계약을 맞췄습니다.”

---

## 7) 한계·확장

**한계**: 정식 `User`/소유권 없음, RAG 계층 없음, 면접은 리포트 중심.  

**확장 후보**: 로그인·소유권, 버전/부분 재생성, (선택) 검색 보강, 면접 코칭 고도화.
