# 이 프로젝트에서의 LangChain 적용 정리

## 1) 왜 LangChain을 사용했는가

이 프로젝트의 목표는 "한 번 생성"이 아니라,  
이력서/포트폴리오/채용공고를 비교하고 보완한 뒤 문서와 면접 준비까지 이어지는 **단계형 워크플로우**를 만드는 것이었다.

단일 프롬프트 호출(one-shot)만으로도 텍스트 생성은 가능하지만, 다음 문제가 있었다.

- 어떤 입력을 기준으로 어떤 결과가 나왔는지 추적이 어렵다.
- 중간 단계(추출/비교/보완)를 분리해 개선하기 어렵다.
- 실패 시 전체를 다시 돌려야 해서 디버깅/운영 비용이 커진다.
- 단계별로 모델을 다르게 쓰는 라우팅을 일관되게 관리하기 어렵다.

LangChain을 쓰면 단계별 프롬프트/파서/실행을 체인 단위로 분리할 수 있어,  
이 프로젝트처럼 "분석 -> 보완 -> 생성" 흐름을 구조적으로 관리하기 쉬워진다.

---

## 2) 이 프로젝트에서 LangChain이 실제로 하는 일

이 프로젝트에서 LangChain은 주로 **체인 구성과 출력 구조화**에 사용된다.

- 프롬프트 템플릿 관리 (`PromptTemplate`)
- 체인 실행 조합 (`RunnableSequence`)
- JSON 파싱/검증 연결 (`JsonOutputParser` + Zod 검증)
- 단계별 체인 분리:
  - 후보자 프로필 추출
  - 채용공고 분석
  - 갭 탐지
  - 보완 질문 생성
  - 문서 생성
  - 문서 리라이트
  - 면접 대비 리포트 생성
- 작업별 모델 라우팅(light/quality) 연동

즉, LangChain은 "모델 종류 자체"보다  
**워크플로우를 단계별로 안전하게 실행/검증/개선하는 프레임워크 역할**을 한다.

---

## 3) 현재 워크플로우와 LangChain 연결

현재 실제 흐름은 아래와 같다.

1. 입력 저장 (이력서/포트폴리오/강조 프로젝트/채용공고)
2. 구조화 추출
   - 후보자 정보 추출
   - 채용공고 요구사항 추출
3. 갭 분석 + 장·단점 스냅샷(`fitAnalysisJson`) 계산
4. 보완 질문 생성 및 사용자 답변 반영
5. 문서 생성/리라이트
6. 면접 대비 리포트 생성
   - 핵심 질문 3
   - 심화 질문 2
   - whyAsked/answerPoints/modelAnswer/caution 포함

이때 LLM 실행의 각 단계는 `WorkflowRun` 로그로 남겨  
모델/폴백/실패 원인을 추적할 수 있게 되어 있다.

---

## 4) 이 프로젝트가 "아닌 것"

이 항목은 과장 방지를 위해 의도적으로 명확히 구분한다.

- 이 프로젝트는 현재 **완전한 RAG 시스템이 아님**
  - 벡터DB 기반 검색-주입 파이프라인을 구현한 상태가 아님
- 이 프로젝트는 현재 **완전한 자율 에이전트 시스템이 아님**
  - 다중 도구 계획/자율 반복 실행/목표 기반 에이전트 런타임을 구현한 상태가 아님

현재 구현은 아래에 더 가깝다.

- 입력 문서 비교 중심의 구조화 워크플로우
- 단계형 추출/분석/보완/생성 파이프라인
- 운영 가능한 로그/재실행/폴백 전략

향후 확장으로는 검색 계층(RAG)이나 더 강한 에이전트형 코칭을 붙일 수 있지만,  
이는 **현재 구현이 아니라 미래 작업**이다.

---

## 5) 그래도 LangChain이 중요한 이유

단일 provider(Gemini 중심)만 써도 LangChain 가치가 사라지지 않는다.

- 워크플로우 오케스트레이션이 명확해진다.
- 단계별 책임이 분리되어 유지보수가 쉬워진다.
- 프롬프트/파서/모델 라우팅을 단계 단위로 독립 개선할 수 있다.
- 실패 지점을 특정하기 쉬워 운영 안정성이 좋아진다.
- 포트폴리오/면접에서 설계 의사결정을 설명하기 쉬워진다.

---

## 6) 포트폴리오/면접에서 이렇게 설명하면 된다

### 짧은 버전
"LangChain을 사용해 취업 지원 워크플로우를 단계형 체인으로 분리했습니다.  
단순 생성이 아니라 추출-갭분석-보완-문서/면접 준비까지 이어지는 구조를 만들고, 단계별 로그와 모델 라우팅을 운영 관점에서 설계했습니다."

### 중간 버전
"이 프로젝트는 one-shot 프롬프트 대신 LangChain 체인으로 후보자 추출, 채용공고 분석, 갭 탐지, 보완 질문, 문서 생성, 면접 리포트를 분리했습니다.  
각 단계는 JSON 파싱 검증을 두어 실패 지점을 명확히 했고, light/quality 모델 라우팅으로 비용과 품질을 분리 운영했습니다.  
또한 `WorkflowRun` 로그로 폴백/실패 원인을 추적하게 만들어 제품 운영성과 설명 가능성을 확보했습니다."

### 기술 버전
"`PromptTemplate + RunnableSequence + JsonOutputParser + Zod` 조합으로 체인을 모듈화했습니다.  
`LangchainWorkflowService`에서 단계별 라우팅과 예외 처리(폴백 포함)를 통합하고, 실행 메타데이터를 `WorkflowRun`에 저장합니다.  
문서 생성/면접 리포트는 구조화 출력으로 관리하며, 프론트는 해당 구조를 그대로 렌더링해 사용자 가독성과 운영 디버깅을 동시에 만족시켰습니다."

## 7) 현재 한계와 확장 방향

### 현재 강점
- 단계형 구조로 디버깅/개선/운영이 용이
- 모델 라우팅과 과호출 방지(락/레이트리밋/재사용) 적용
- 분석 결과와 실행 로그를 DB에 저장해 재실행/추적 가능

### 현재 한계
- 인증/사용자 소유권(`userId`)은 아직 미구현
- 검색 기반 근거 보강(RAG) 계층은 미구현
- 면접 코칭은 리포트 중심이며 상호작용 코칭은 제한적

### 확장 후보
- 사용자 히스토리 영속화 (로그인 + 소유권)
- save-first/버전 관리/부분 재생성
- 향후 검색 계층 추가(RAG 확장)
- 면접 코칭 고도화(맞춤 피드백, 추적형 개선)
