# CareerFlow AI 실측 테스트 및 문제 해결 기록

## 왜 이 테스트를 했는가

CareerFlow AI는 일반 CRUD 서비스와 다르게, 입력 저장 이후에 LLM이 포함된 다단계 워크플로우가 이어집니다.
따라서 이 프로젝트의 성능을 보여줄 때도 단순 TPS 숫자 하나만 제시하는 방식보다 아래 세 가지를 함께 확인하는 편이 더 적절했습니다.

- 입력 저장 구간은 어느 정도의 처리량을 버틸 수 있는가
- 분석, 문서 생성, 면접 생성 같은 고비용 구간은 얼마나 오래 걸리는가
- 중복 실행, 재생성, 과호출 상황에서 비용과 시간을 실제로 제어하고 있는가

이번 기록은 포트폴리오용 설명이 아니라, 실제 로컬 환경에서 API를 직접 호출해 측정한 결과를 기준으로 작성했습니다.

- 서버 실행 방식: NestJS dev server

- LLM provider: Gemini
- Node.js: `v20.20.1`
- 장비:
  - CPU: AMD Ryzen 5 5600 6-Core Processor
  - Memory: 16 GB
- 아티팩트:
  - [test-artifacts/careerflow-benchmark-2026-04-14T18-14-49-877Z.json](./test-artifacts/careerflow-benchmark-2026-04-14T18-14-49-877Z.json)

주의:

- 이 결과는 로컬 개발 환경 기준 실측값입니다.
- 따라서 운영 서버의 절대 TPS로 해석하기보다, 워크플로우의 상대적인 병목과 제어 설계를 검증한 기록으로 보는 편이 맞습니다.

## 테스트 시나리오

### 1. 입력 저장 API 기본 처리량 확인

`POST /source-documents`

- 순차 6회 호출
- 동시 12회 호출
- LLM 없이 DB 저장만 일어나는 구간이라, 시스템의 가벼운 ingress 처리 능력을 확인하기 위한 목적

### 2. 분석 워크플로우 전체 실행 시간 측정

`POST /analysis/run`

- 실제 이력서, 포트폴리오, 공고 텍스트를 넣고 분석 전체 실행
- `PARSE_SOURCE -> EXTRACT_CANDIDATE -> EXTRACT_JOB -> DETECT_GAP -> GENERATE_FOLLOW_UP`
- 단계별 fallback 여부와 사용 모델도 함께 확인

### 3. 동일 워크플로우 동시 실행 충돌 확인

`POST /analysis/run` 동일 `applicationId` 동시 호출

- 첫 번째 요청은 정상 실행
- 두 번째 요청은 lock에 걸려 즉시 `409 Conflict`가 나와야 함

### 4. follow-up 중복 제출 처리 확인

`POST /followup-questions/submit`

- 첫 제출 후, 동일 answers를 한 번 더 제출
- 의도는 두 번째 요청이 재실행되지 않고 skip 처리되는지 확인하는 것

### 5. 문서 생성 재사용 및 레이트리밋 확인

`POST /generated-documents/generate`

- 첫 호출은 실제 생성
- 이후 동일 요청은 기존 결과를 재사용하며 skip
- 짧은 시간에 반복 호출하면 `429`로 차단되는지 확인

### 6. 면접 리포트 재사용 및 레이트리밋 확인

`POST /interview/generate`

- 첫 호출은 실제 생성
- 이후 동일 요청은 기존 결과를 재사용하며 skip
- 반복 호출 시 `429` 발생 확인

## 실측 결과 요약

### 입력 저장 구간

- 순차 6회 평균: `16.2ms`
- 순차 6회 p95: `57.6ms`
- 동시 12회 배치 시간: `235.7ms`
- 동시 12회 추정 처리량: `50.92 req/s`
- 동시 12회 평균 응답: `136ms`

해석:

- LLM이 없는 저장 구간은 비교적 가볍고 빠르게 응답했습니다.
- 이 프로젝트는 진입 구간이 아니라, 이후 AI 워크플로우 단계가 전체 체감 속도를 결정합니다.

### 분석 워크플로우

- 전체 실행 시간: `86,653.9ms` (`약 86.7초`)
- 최종 상태: `ANALYZED`
- 기록된 모델:
  - `gemini:gemini-2.5-pro`
  - `gemini:gemini-2.5-flash`
  - `gemini:gemini-2.5-flash-lite`
- fallback 발생: `0회`

해석:

- 실제 병목은 CRUD가 아니라 LLM 단계형 실행에 있었습니다.
- 다만 이번 측정에서는 fallback 없이 전 단계가 완료되어, 라우팅과 structured output 제어가 안정적으로 동작하는 것을 확인했습니다.

### 동일 워크플로우 동시 실행 충돌

- 1차 요청: 정상 완료 `78,626.8ms`
- 2차 요청: `409 Conflict`, `2.8ms`
- 에러 메시지:
  - `Workflow stage is already running: analysis:59`

해석:

- 동일 `applicationId`에 대한 중복 실행이 즉시 차단되었습니다.
- 비싼 분석 요청이 겹쳐서 중복 과금되거나 DB 상태가 꼬이는 위험을 실제로 줄이고 있음을 확인했습니다.

### follow-up 제출

초기 측정:

- 첫 제출: `4,124.3ms`
- 중복 제출: `4,532.5ms`

이상 징후:

- 중복 제출은 원래 skip되어야 했는데, 실제로는 첫 요청과 비슷한 시간대가 걸렸습니다.
- `WorkflowRun`을 확인하니 `SKIPPED_DUPLICATE_FOLLOWUP_ANSWERS`가 아니라, `REGENERATE_CANDIDATE`가 다시 실행되고 있었습니다.

원인:

- 기존 로직이 아래처럼 단순 문자열 비교에 의존하고 있었습니다.

```ts
JSON.stringify(app.followUpAnswersJson ?? []) === JSON.stringify(dto.answers)
```

- JSON 객체는 key 순서나 직렬화 형태 차이로 인해, 의미상 동일한 데이터라도 문자열 비교에서는 다르게 판정될 수 있습니다.
- 실제로 저장된 JSON과 요청 payload의 key 순서 차이 때문에 duplicate skip이 실패했습니다.

적용한 수정:

- answers를 `questionId`, `answer` 기준으로 trim + normalize
- 정렬된 형태로 비교
- 저장할 때도 normalized payload를 그대로 사용

수정 후 재검증:

- 중복 제출 응답 시간: `32.9ms`
- 마지막 `WorkflowRun.errorMessage`:
  - `SKIPPED_DUPLICATE_FOLLOWUP_ANSWERS`

해석:

- 원래는 4초대 재실행으로 LLM 호출이 새고 있었고,
- 수정 후에는 수십 ms 안에 스킵되면서 비용과 대기 시간을 동시에 줄였습니다.

이 부분은 이번 테스트에서 가장 의미 있었던 실제 문제 해결 사례였습니다.

### 문서 생성

- 첫 생성: `26,378.5ms`
- 동일 요청 재호출: `6.1ms`
- 반복 호출 후 레이트리밋 응답: `429`, `2.7ms`

관찰:

- 첫 실행은 실제 생성 비용이 반영되어 수십 초가 걸렸습니다.
- 이후에는 기존 초안을 재사용하며 매우 빠르게 응답했습니다.
- 짧은 시간에 과도한 재호출이 들어오면 즉시 차단되었습니다.

### 면접 리포트 생성

- 첫 생성: `26,362.6ms`
- 동일 요청 재호출: `6.0ms`
- 반복 호출 후 레이트리밋 응답: `429`, `1.7ms`

관찰:

- 문서 생성과 유사하게, 생성 비용이 큰 구간과 재사용 가능한 구간이 명확히 분리되어 있었습니다.
- 결과가 이미 존재할 때 불필요한 재생성을 막는 구조가 실제로 효과를 내고 있음을 확인했습니다.

## 이번 테스트에서 확인한 강점

### 1. 비싼 구간을 정확히 보호하고 있다

- 동일 단계 동시 실행은 `409`로 즉시 차단
- 문서/면접 재생성은 skip 처리
- 짧은 시간 반복 호출은 `429` 차단

즉, 무거운 LLM 구간을 아무 생각 없이 다시 태우지 않도록 기본 보호막이 잘 들어가 있었습니다.

### 2. LLM 실행 메타데이터가 남아 원인 추적이 가능하다

`WorkflowRun.inputJson.llmRoute`, `llmExecution`, `fallbackReason` 구조 덕분에 다음 질문에 답할 수 있습니다.

- 어느 단계에서 어떤 모델을 썼는가
- fallback이 발생했는가
- 실패가 quota인지, provider 이슈인지, 파싱 이슈인지

이건 단순히 “AI 기능을 붙였다”보다 운영 가능한 워크플로우에 더 가깝습니다.

### 3. 저장 구간과 AI 구간의 병목이 명확히 분리되어 있다

- 입력 저장은 빠르고
- 실제 병목은 분석/문서/면접 생성입니다.

이 구분이 명확해야 이후 최적화도 헛돌지 않습니다.

## 이번 테스트에서 드러난 한계

### 1. 전체 분석 시간은 여전히 길다

`analysis/run` 전체가 약 86초 수준이라, 사용자 체감상 즉각적인 서비스는 아닙니다.

이 프로젝트의 현재 목표는 실시간 챗봇보다 “단계형 문서 준비 지원”에 가깝기 때문에 치명적인 문제는 아니지만, 이후에는 아래 개선을 고려할 수 있습니다.

- 단계별 비동기 큐 분리
- 이미 유효한 중간 산출물 재사용
- 문서 생성/면접 생성 lazy trigger 강화
- stage별 elapsed time 저장

### 2. 상태값만 보면 마지막 단계가 덜 보인다

현재 `Application.status`는 문서 생성 이후 `DOCUMENTS_GENERATED`에서 머무르고, 면접 리포트 생성 완료가 상태값에 반영되지는 않습니다.

실제 UI는 `hasInterviewPrep`로 보완하고 있지만, 워크플로우 상태만 따로 볼 때는 마지막 단계 가시성이 약합니다.

이 부분은 다음 개선 후보입니다.

## 포트폴리오에 넣을 수 있는 설명 문장

### 짧은 버전

- CareerFlow AI는 단순 생성형 기능이 아니라, 분석·보완·문서 생성·면접 대비까지 이어지는 단계형 AI 워크플로우로 설계했습니다.
- 실제 API 테스트를 통해 동시 실행 충돌, 중복 재생성, 과호출 상황을 검증했고, lock·skip·rate limit 구조가 비용과 응답 시간을 줄이는 것을 확인했습니다.
- 테스트 과정에서 follow-up 중복 제출이 재실행되는 버그를 발견했고, JSON 직렬화 비교 로직을 normalized comparison으로 수정해 4초대 재실행을 32.9ms skip 처리로 개선했습니다.

### 면접용 버전

- 이 프로젝트에서 가장 중요하게 본 것은 단순 응답 속도보다, 비싼 LLM 호출을 얼마나 통제 가능한 흐름으로 묶는가였습니다.
- 실제로 `analysis/run` 동시 호출 실험을 통해 동일 워크플로우 중복 실행이 `409`로 즉시 차단되는지 확인했고, 문서/면접 생성은 기존 결과가 있을 때 수 ms 수준으로 skip되도록 검증했습니다.
- 테스트 중 follow-up answers duplicate check가 문자열 비교 방식이라 재실행이 새는 문제를 발견했고, 입력을 normalize해서 비교하도록 바꾼 뒤 중복 제출이 32.9ms 안에 `SKIPPED_DUPLICATE_FOLLOWUP_ANSWERS`로 처리되는 것을 확인했습니다.

## 다음 단계 제안

이번 기록으로도 포트폴리오에 넣을 만한 근거는 충분하지만, 한 단계 더 올리려면 다음 순서가 좋습니다.

1. `WorkflowRun`에 단계별 elapsed time을 남겨 stage 병목을 더 정확히 추적
2. 배포 환경(`t3.small` 또는 `t3.medium` 등)에서 k6/Artillery 기반 원격 부하 테스트 추가
3. 결과를 포트폴리오 프로젝트 상세에 `Problem -> Test -> Fix -> Result` 형식으로 재배치

지금 기준으로는 이미 “무엇을 만들었다” 수준을 넘어서, “실제 실행 흐름을 검증하고 개선한 프로젝트”라고 설명할 수 있습니다.
