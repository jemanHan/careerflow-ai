# Next Roadmap

## 목적
- 현재 MVP를 "데모 가능" 수준에서 "지속 사용 가능한 제품" 수준으로 확장
- 비용/안정성/개인화/보안/포트폴리오 증빙을 함께 강화

## 현재 기준선
- Gemini 기반 워크플로우(분석/후속질문/문서/면접질문/리라이트) 동작
- `Application`, `WorkflowRun`에 단계별 결과와 실행 메타데이터 저장
- 결과 페이지 UX(근거 요약/질문별 답변/로딩 상태) 개선 완료
- 라이트 테스트 계정(`TestUser`)으로 저장 워크플로우 재방문 가능(데모용)

## 다음 단계 (우선순위)

### P0. 정식 사용자 계정/개인화 기반
- 현재 라이트 테스트 계정 흐름을 정식 인증으로 확장
- 로그인 기능 추가 (세션 또는 JWT)
- `Application`을 사용자(`userId`)와 연결
- 사용자별 조회/수정 권한 체크(본인 데이터만 접근)
- 기대효과:
  - "저장형 취업 워크플로우" 전환
  - 재방문 시 누적 개선 가능
  - 사용자별 데이터 분리로 개인정보/권한 관리 기반 확보

### P0. 저장 중심 사용 흐름
- 매번 전체 재생성 대신 부분 재실행(분석만/문서만/면접만)
- 버전 스냅샷 저장 (초안 v1, v2...)
- 후속 답변 전/후 비교 뷰 추가
- 기대효과:
  - LLM 호출 비용 절감
  - 사용자 체감 품질 개선
  - 원샷 생성이 아닌 반복 개선형 취업 준비에 적합

### P1. 보안/개인정보 보호
- 이력서/포트폴리오 민감정보 저장 정책 명시
- 삭제 기능(사용자 데이터 완전 삭제)
- 로그 민감정보 마스킹
- 운영 문서에 보관기간/처리방침 초안 추가

### P1. 품질 고도화
- 문서 톤 프리셋(보수적/균형/강조형)
- 경력기술서 초안 포맷 고정 템플릿
- 후속 답변 점수 서버 저장 + 개선 가이드 자동 제시
- "과장 위험 문장" 룰 기반 후처리

### P1. Retrieval-assisted 확장 (RAG-like, 단계적)
- 현재 구현은 **구조화 워크플로우 + 저장/재실행**이며, full RAG(벡터DB 검색 주입 파이프라인)는 아직 구현하지 않음
- 목표는 "지금 있는 저장 자산"을 검색해 다음 생성 품질을 높이는 **retrieval-assisted 진화**
- 확장 전제: 직군 고정 버킷이 아닌 JD 요구 신호/근거 신호 비교를 기본 축으로 유지
- 1차는 경량 검색부터:
  - 사용자 기술 기반 공고 탐색 키워드 추천(최대 1~3개)
  - 사용자 저장 이력서/포트폴리오/보완답변 히스토리
  - 이전 생성 결과(문서 초안/면접 리포트) 재참조
  - 직무별 템플릿/예시 라이브러리
- 2차는 선택적으로 RAG 파이프라인 확장:
  - 임베딩 인덱싱 + 관련 스니펫 검색 + 프롬프트 컨텍스트 주입
  - 단계별(장·단점 분석/문서/면접) 다른 검색 소스 조합
- 기대효과:
  - 장·단점 분석: 입력 단일 시점 평가에서 "누적 근거 기반" 요약으로 개선
  - 문서 생성: 과거 강점 문장/성과 근거 재사용으로 품질·일관성 향상
  - 면접 준비: 이전 약점/주의 포인트를 반영한 맞춤 질문 고도화
  - 제품 가치: 재방문할수록 좋아지는 경험 강화, 개인화 고착도 상승, 향후 유료화(저장량/고급 검색) 설계 근거 확보

### P1. 운영 지표 대시보드
- 단계 성공률, fallback율, 평균 처리시간, 호출량 지표화
- 최근 7일 기준 성공률 카드 추가
- 장애 유형(키/쿼터/파싱) 비율 자동 집계

### P2. 배포/운영 완성
- 배포 환경 분리(dev/stage/prod)
- 에러 알림(간단 웹훅 또는 로그 수집)
- 백업/복구 시나리오 문서화
- 데모 URL + 포트폴리오 캡처 세트 정리

## 실행 순서 제안 (2주 샘플)
- Day 1-3: 정식 로그인 + userId 연결 + 권한 체크
- Day 4-5: 저장/버전/부분 재실행
- Day 6-7: 보안/삭제/마스킹
- Day 8-10: 품질 튜닝(톤/템플릿/후처리)
- Day 11-14: 운영 지표 + 배포 마무리

## 예상 스키마 확장(계획)
- `User`
  - `id`, `email`, `passwordHash`(또는 OAuth 식별자), `createdAt`
- `Application.userId`
  - 사용자 소유권 연결(FK)
- `DraftVersion` (선택)
  - `applicationId`, `kind`(cover/career/project/interview), `version`, `content`, `createdAt`
- `RegenerationJob` (선택)
  - 부분 재생성 요청 이력, 실행 상태, 대상 단계

## 범위 명확화
- 이 문서는 **향후 방향** 문서이며, 현재 MVP 범위에 로그인/권한 기능은 포함되지 않는다.
- 현재 구현은 비로그인 + 라이트 테스트 계정 기반 저장/재조회 흐름까지 제공한다.
- 현재 구현은 full RAG가 아니며, 공고 탐색 키워드 추천은 경량 참고 기능이다.

## 향후 확장 방향: 로그인 기반 개인화 및 저장형 워크플로우
- 현재 MVP는 개인 개발/단기간 검증 우선순위로 로그인·인증을 의도적으로 제외했다.
- 대신 핵심 가치(입력 구조화, 공고 대비 장·단점 분석, 보완 질문, 문서 생성, 면접 리포트)를 먼저 구현해 제품 가설을 검증했다.
- 현재 아키텍처는 PostgreSQL 영속화(`Application`)와 단계 로그(`WorkflowRun`)를 이미 갖추고 있어, 이후 `userId` 연결만 추가하면 사용자별 저장/개인화로 자연스럽게 확장 가능하다.
- 이 구조는 취업 준비의 반복 개선 특성(재수정/부분 재생성)에 맞고, 기존 분석·생성 결과 재사용으로 불필요한 LLM 재호출을 줄여 비용 통제와 장기 운영성에 유리하다.
- 향후 확장 후보:
  - 로그인/인증
  - 사용자별 데이터 격리
  - 입력/보완/생성 결과 버전 관리
  - 부분 재생성 및 변경 이력 추적
  - 사용량 기반 운영 및 비용 최적화(향후 수익화 기반)

## 포트폴리오 증빙 포인트
- "한 번 생성"이 아니라 "누적 개선" 제품 흐름 설계
- AI 비용 절감(재사용/부분 재실행)과 안정성(락/레이트리밋) 동시 달성
- 개인정보 저장 제품에서 필요한 최소 보안/권한/삭제 정책 적용
